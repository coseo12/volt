## 배경/상황

P1·P2 두 Phase를 스프린트 계약(사전 합의된 검증 가능 완료 기준) + 마일스톤 회고 루틴으로 진행. 24개 이슈 × 5 마일스톤(P2)을 AI 혼자 과잉 긍정 평가하지 않고 일관되게 완료.

## 내용

**스프린트 계약 구조** (CLAUDE.md에 템플릿):
1. Phase/마일스톤 착수 전 **검증 가능한 완료 기준 목록** 합의.
2. 각 기준은 **측정 가능**해야 함 (예: "< 0.1% 오차", "60fps", "axe 0 위반", "N=1000 PASS").
3. 미충족 시 **원인 + 구체적 수정점**으로 반려 (단순 "실패" 금지).
4. 표면 테스트가 아닌 **엣지 케이스 포함**.

**회고 루틴** (마일스톤 종료마다 필수):
1. `docs/retrospectives/<phase>-retrospective.md` 생성.
2. 섹션: 달성도 표 / 잘 된 것 / 어려웠던 것 / 프로세스 교훈 / 다음 인수인계.
3. 테스트 증분, 성능 변화 수치를 baseline 대비 기록.

**실측 효과**:
- P1에서 드러난 프로세스 사고(UI 검증 누락, Vitest 누락)가 P2-0에서 가드(PR 템플릿, verify:test-coverage)로 제도화 → P2 전체 재발 0건.
- 회고마다 P2-C/D 인수인계 섹션이 다음 마일스톤 이슈 분해의 씨앗이 됨.
- 사용자가 "AI 과잉 긍정 평가" 경향을 사전 합의된 기준으로 통제.

**이터레이션**:
- 초기 회귀 기준(P2-B ±2fps)은 콘텐츠 추가 후 유지 불가 → 25%로 완화 + baseline 갱신.
- 기준은 실측 후 조정 가능해야 함. 고정이 아니라 "합의된 재조정 가능".

## 교훈 / 다음에 적용할 점

- **모든 Phase/마일스톤 착수 전 "검증 가능한 완료 기준" 체크리스트를 사용자와 합의한다.** 조건이 측정 불가능하면(모호하면) 분해해서 다시 정의.
- **마일스톤 종료 시 retrospective 파일은 의무.** 테스트 증분·성능 변화·교훈·다음 인수인계 4섹션 고정.
- skill: `sprint-contract` / `retrospective` 같은 템플릿을 제공해 agent가 일관되게 생성.
- 사용자와의 합의 지점이 명확할수록 AI 과잉 긍정 평가가 억제됨 — 애매한 지시("리뉴얼", "개선")는 범위 확정 질문이 기본값.
