볼트 자체 운영 중 관찰되는 신호를 날짜별로 누적. 자동화·구조·규약 변경 판단의 시계열 근거.

각 엔트리 구조:
- **처리량** — 이번 배치에서 정제한 이슈/노트 수
- **관찰** — 흐름에서 눈에 띈 것(마찰·품질·일관성)
- **시그널** — 앞으로의 방향에 주는 함의

## 2026-04-14 — 초기 구축 + 1차 배치(#1~#8)

**처리량**: 8건 (feedback 2 / pattern 3 / decision 1 / knowledge 2)

**관찰**:
- 볼트 구조 설계 + 레포 생성 + 문서(AGENTS.md / CONTRIBUTING.md) + 초기 이슈 8건 정제를 같은 날 완수.
- 대부분 이슈가 이미 잘 구조화돼 있어 `inbox/` 단계에서 본문 교정은 거의 없음.
- 예외: #8(`/volt` 슬래시커맨드 가이드)에서 "볼트 파서" 표현 및 `knowledge/`, `reports/` 폴더명이 실제 구조(`notes/`, `meta/*`)와 불일치 → 정제 시 교정.
- 분류 분포: `meta/patterns/` 가 3건으로 가장 많음. 개발 경험이 "재사용 가능한 패턴" 형태로 자주 추출됨.
- `feedback` 2건 모두 "빌드 PASS = 동작 PASS 착각" 계열 — 공통 가드 설계 여지.

**시그널**:
- 이슈 본문 품질이 안정적이라면 자동화의 "내용 교정" 부담은 작음 → Tier 1(이슈→inbox 변환) 가치 상승.
- 자기 참조 문서(#8처럼 도구 자체를 문서화)에서는 구조 드리프트 위험 큼. 정제 단계 교정이 필수.
- `meta/patterns/` 편중 → RAG 인덱싱 우선순위 후보.

## 2026-04-15

### 2차 배치(#9~#11)

**처리량**: 3건 (knowledge 2 / report·troubleshooting 1) — 모두 `notes/` 로

**관찰**:
- 세 건 모두 본문 교정 없이 원문 그대로 정제. 캡처 품질 안정 유지.
- 출처 PR 링크가 모두 포함됨 → 역추적 가능성 높음.
- 세 건 모두 같은 출처(astro-simulator) + P3-B 시점에 집중 → 프로젝트·시기별 버스트 패턴.
- #11은 troubleshooting 이지만 재사용 가능한 "bench 함정" 교훈 성격 → `notes/` 유지하되 나중에 태그로 패턴성 필터링 필요.

**시그널**:
- 캡처 품질 안정 → 자동화 시 "내용 그대로 옮기기" 모드가 대부분 케이스 충족. 교정은 예외 처리로 충분.
- 같은 출처 레포에서 버스트로 올라오는 경향 → `source_repo` 별 인덱스/필터 유틸이 나중에 유용할 수 있음.
- 이슈 템플릿의 폼이 잘 지켜지고 있음 (헤더명 일탈 0건) → 파서 기반 자동화의 기술적 전제가 성립.

### 3차 배치(#12~#13)

**처리량**: 2건 (knowledge 1 / report·troubleshooting 1) — 모두 `notes/` 로

**관찰**:
- 두 건이 동일 PR(astro-simulator #163)에서 파생된 쌍둥이 교훈(#12 구조 패턴 + #13 함정 회피). frontmatter `related` 에 상호 참조 추가 — 관련 노트 연결의 첫 사례.
- **헤더 포맷 변형 첫 감지**: 두 이슈 본문이 `### 헤더\n값` (빈 줄 생략) 형태로 작성됨. 지금까지는 GitHub 이슈 폼이 생성하는 `### 헤더\n\n값` 형태만 있었음. 헤더 *이름* 규약은 유지, *포맷* 만 변형. 수동 파싱에선 문제 없지만 파서 기반 자동화 설계 시 두 포맷 모두 허용해야 함.
- 본문 교정 없이 원문 그대로 정제.

**시그널**:
- 관련 노트 상호 참조가 RAG 맥락 유지에 유의미 → 같은 PR/이슈/주제에서 파생된 쌍은 `related` 에 서로 포함하는 패턴으로 정착.
- 파서 자동화의 "기술적 전제"가 아주 작게 금이 감지됨 — 헤더 이름은 여전히 0 일탈이지만 주변 공백 허용 설계가 필수가 됨.
- 누적 집계 표에 "헤더 포맷 변형" 지표 신설.

### 후속 보강: #13 (closed 이후 코멘트로 추가)

**처리량**: 노트 1건 보강 (신규 캡처 아님, 기존 `notes/lint-staged-gitignore-silent-revert.md` 에 섹션 추가)

**관찰**:
- **정제 후 보강 워크플로 첫 사례**: 이슈가 close 된 뒤에도 동일 주제의 추가 관찰이 코멘트로 유입됨. 기존 가정("close = 정제 종료")이 현실과 어긋남. 노트는 정지된 문서가 아니라 살아있는 문서로 동작.
- 대응: 노트에 `## 보강 (YYYY-MM-DD 후속)` 섹션을 추가하고, closed 이슈에 갱신 커밋 코멘트를 남기는 방식으로 처리. 재오픈은 하지 않음.
- `git add -f` 우회책 검증, 배포 구조 관점 근본 해결안 2종, harness 가드 우선순위 격상 메모 등이 담김.

**시그널**:
- **AGENTS.md / CONTRIBUTING.md 에 "사후 보강 절차" 공식화 필요**: closed 이슈 코멘트 → 담당자가 노트에 보강 섹션 추가 + 이슈에 커밋 코멘트 회신. 지금은 관행으로 돌아가지만 여러 사람/세션이 관여하게 되면 규약 명문화가 유리.
- **frontmatter 에 `last_reinforced_at` 필드 도입 검토**: 보강이 반복되면 "최근 갱신일"을 메타데이터로 노출하는 것이 RAG 신선도 판단에 유용할 수 있음 (지금 도입은 이르다고 보고 보류).
- 누적 집계 표에 "노트 보강 건수" 지표 신설.

## 2026-04-16 — 4차 배치(#14~#18)

**처리량**: 5건 (knowledge 4 / report·troubleshooting 1) — 모두 `notes/` 로

**관찰**:
- **최대 배치 규모 갱신 (단일 세션 5건)**. 이전 최대는 2026-04-14 초기 구축일의 8건이었으나 그건 구축+정제 혼합. 순수 정제 세션 기준 최대치.
- **헤더 포맷 표준형 복귀**: 5건 모두 `### 헤더\n\n값` 정규형. 3차 배치(#12, #13)의 빈 줄 생략 변형은 **일회성 사례로 확정** — 이슈 폼 경로가 기본값임을 재확인.
- **related 그래프 밀도 상승**: 단순 쌍(#12↔#13) → 삼각 구조(#15↔#16, #16↔#18, #15→#18 벤치 삼각) + 기존 노트 역참조 첫 사례(#14 → `meta/patterns/sprint-contract-retrospective.md`).
- 본문 교정 없이 원문 그대로 정제. 품질 안정 유지.
- 여전히 단일 출처(astro-simulator) — 버스트 주기가 astro-simulator 마일스톤 종료에 맞춰짐(P3-B, P4 후속).
- 분류 경향: knowledge 라벨이 "패턴성 지식"까지 포괄하는 실사용이 반복됨(#14, #18). `notes/` 내 태그로 패턴성 필터링 검토는 여전히 보류 가능.

**시그널**:
- 세션당 처리량이 5건까지 확장되어도 마찰 없이 처리됨 → 수동 정제의 한계는 아직 보이지 않음.
- 헤더 포맷 변형은 **이슈 폼을 우회한 수동 작성 시에만 발생**. 즉 파서 자동화 설계 시 "폼 경로 기본형 + 수동 경로 유연 허용" 이중 전략으로 충분.
- related 그래프가 3노드 이상으로 확장되어 RAG 맥락 검색의 jump 가치 상승. 관계 유형을 구분(같은-주제 / 보강 / 대안)할 필요는 **아직 없음** — 단순 양방향 리스트로 유지.
- 출처 레포 다양화 시그널 여전히 미발생. 1종 지속 시 "source_repo 별 인덱스 유틸" 도입 시점 재평가 필요.
